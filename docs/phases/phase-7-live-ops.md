# Phase 7 — 라이브 운영 (차별점 실행)

> 목표: 기획의 차별점 — "방치형에 라이브 서비스를 1급 시민으로".
> 규모: 큼, 단 기능별 독립 배포 가능. 선행: Phase 6 (유저와 지표가 있어야 의미 있음)
> **순서는 소프트 런칭 피드백으로 재조정한다.** 아래는 구현 난이도·의존성 기준 권장 순서.

## 0. 공통 전제
- 모든 라이브 기능은 Phase 4의 **세이브 검증**을 신뢰 기반으로 삼는다 (검증 없는 랭킹은 무의미)
- 서버 권위 원칙: 보상 지급·순위 계산은 전부 서버. 클라는 표시만
- 기능 플래그: `server`에 기능 on/off 환경 변수 — 문제 시 게임 중단 없이 기능만 끔

## 7-1. 주간 월드 보스 (첫 라이브 콘텐츠 — 가장 단순)

서버 전체가 일주일간 보스 HP를 함께 깎는다. 기여도 비례 보상.

### 설계
- 보스 HP는 천문학적 수치 (전원이 깎아도 다 못 깎을 수 있음 — 처치 시 전 서버 보너스)
- 기여 방식: 플레이어가 "월드 보스 토벌" 작업을 슬롯에 배정 →
  동기화 시점에 서버가 `전투력 × 경과시간 = 누적 딜`을 **서버 계산**으로 적립
  (클라가 딜 수치를 보고하지 않는다 — 상태에서 서버가 도출)
- 주간 리셋: cron (일요일 자정 KST) — 순위 확정 → 보상 메일함 지급 → 새 보스
- 보상: 기여 구간별 (참여만 해도 소액) + 상위 % 추가. 경쟁과 참여 동기 양립

### 작업
- [ ] DB: `world_boss(week, bossId, totalHp, remainingHp)`, `boss_contribution(week, userId, damage)`
- [ ] 메일함 시스템 (보상 수령 UI) — 이후 모든 라이브 보상의 공통 인프라. **먼저 만든다**
- [ ] 토벌 작업 타입 추가 (코어) + 서버 적립 로직 + cron 정산
- [ ] UI: 보스 패널 (남은 HP 게이지, 내 기여, 실시간 순위 상위 10)

### DoD
- [ ] 두 계정으로 기여 → 주간 정산 → 기여 비례 보상이 메일함에 도착

## 7-2. 랭킹

### 설계
- 종목: 스킬별 XP, 총 레벨, 주간 월드보스 딜. 시즌 도입 후엔 시즌별로 분리
- Redis sorted set (zadd 동기화 시점, zrevrange 조회) + 일 1회 Postgres 스냅샷
- 표시: 상위 100 + 내 순위. 닉네임 필요 → **닉네임 설정 기능이 선행 작업** (중복·금칙어 검사)

### 작업
- [ ] Redis 추가 (Fly.io/Upstash 무료 티어)
- [ ] 닉네임 설정 (User.nickname, 변경 쿨다운)
- [ ] 동기화 성공 시 zadd, `GET /rankings/:board` API, 랭킹 패널 UI

### DoD
- [ ] 세이브 검증을 통과한 진행만 순위에 반영된다

## 7-3. 거래소 (비동기 아이템 매매)

### 설계 — 경제 안정장치를 먼저 정한다
- 모델: 등록(아이템 에스크로) → 구매(즉시) → 정산. 입찰 없음 (단순)
- **수수료 5~8% (골드 소각)** — 인플레이션 싱크. 등록 수수료도 소액 (스팸 방지)
- 가격 제한: 기준가(NPC 판매가)의 0.5×~20× — 골드 세탁/실수 방지
- 거래 불가 품목 분리: 던전 핵심 재료 등은 거래 불가 태그 (진행 스킵 방지)
- 모든 거래 로그 보존 (이상 거래 추적)

### 작업
- [ ] DB: `listing(id, sellerId, itemId, qty, price, status, createdAt)` + 거래 로그
- [ ] 코어: 아이템 `tradable` 태그
- [ ] API: 등록/취소/구매/내 판매 현황 (구매는 트랜잭션 — 동시 구매 경합 처리)
- [ ] UI: 거래소 패널 (검색, 시세 = 최근 체결가)
- [ ] 운영 쿼리: 일일 골드 발행량 vs 소각량 대시보드

### DoD
- [ ] 두 계정 간 거래 왕복 + 수수료 소각 확인
- [ ] 동시 구매 경합에서 한 명만 성공한다

## 7-4. 시즌 리그 (가장 크고 마지막)

### 설계
- 기간: 2~3개월. 시즌 캐릭터 = 본 계정과 분리된 새 GameState (Save에 `seasonId` 차원 추가)
- 시즌 룰 = 차별화 요소 (예: "이번 시즌은 채집 속도 2배, 거래소 없음") — 룰은 서버 설정으로
- 시즌 종료: 시즌 랭킹 확정 → 보상 = **본 계정 영구 치장/칭호 + 소량 부스트** (P2W 금지)
  → 시즌 캐릭터는 아카이브 (조회만 가능)
- 신규 유저에게도 공평한 출발선이라는 점이 핵심 가치 (PoE 리그 모델)

### 작업
- [ ] Save 모델 다중화 (userId + seasonId 복합키), 시즌 메타 테이블
- [ ] 시즌 룰 주입: 코어 simulate에 `modifiers` 파라미터 (배율 등) — 코어 변경 신중하게
- [ ] 시즌 전환 cron + 보상 지급 (메일함 재사용)
- [ ] UI: 시즌 선택 (본 서버/시즌), 시즌 정보·남은 기간, 칭호 표시

### DoD
- [ ] 시즌 시작→플레이→종료→보상→아카이브 전체 사이클이 테스트 시즌(짧은 기간)으로 검증됨

## 리스크 / 주의
- **거래소는 게임 경제를 한 번에 망칠 수 있는 유일한 기능** — 수수료·가격 제한·로그 없이 열지 말 것.
  문제 발생 시 기능 플래그로 즉시 닫는다
- 월드보스→랭킹→거래소→시즌 순서는 의존성 순서이기도 하다 (메일함→닉네임→경제→전체 구조)
- 시즌 룰의 코어 `modifiers`는 결정성·검증과 충돌하지 않게 설계 리뷰 후 구현
- 여기까지 오면 운영 부담이 실재한다 — 주 1회 정기 점검 시간을 캘린더에 고정할 것
