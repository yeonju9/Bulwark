# Phase 3 — 스킬 확장 (경제 순환 완성)

> 목표: 스킬끼리 서로의 산출물을 소비하는 경제 순환. "팔기만 하는 아이템 없음".
> 규모: 중간 (시스템 < 콘텐츠 작업). 선행: Phase 2 (음식 슬롯·마정석 드랍이 전제)

## 1. 설계 결정

### 1-1. 경제 순환도 (목표 상태)
```
벌목 ──목재──→ 대장기술(자루)            채광 ──광석──→ 대장기술 ──장비──→ 전투
낚시 ──생선──→ 요리 ──음식──→ 전투(회복)   전투 ──가죽──→ 대장기술(방어구)
약초(채집 부산물) + 마정석(전투) ──→ 연금술 ──물약──→ 채집 버프 / 전투 버프
모든 스킬 ──골드──→ 상점(도구 업그레이드) ──→ 채집 속도
```
설계 검증법: 모든 아이템에 대해 "소비처가 어디인가"를 표로 만들었을 때 빈 칸이 없어야 한다.

### 1-2. 버프(물약) 시스템
- 물약 = 소비 시 일정 시간 효과 (`예: 30분간 벌목 속도 +15%`)
- 동시 적용 1종 제한 (카테고리별 1개: 채집물약/전투물약) — 스택 복잡도 차단
- 효과는 `cycleDurationMs`/`computeStats`에 곱연산으로만 개입 (가산 금지, 밸런스 단순화)

### 1-3. 상점 (골드 싱크)
- 영구 도구 업그레이드: `청동 도끼 → 철 도끼 → …` 스킬별 채집 속도 -5%/단계
- 가격은 지수 증가. 골드 인플레이션의 1차 싱크 (Phase 7 거래소 수수료가 2차)

## 2. 데이터 모델 변경 (세이브 v4)

```ts
export type SkillId = ... | 'fishing' | 'cooking' | 'alchemy';  // 8종이 됨

// GameState 추가
buffs: ActiveBuff[];          // { buffId, expiresAtMs(게임시간 누적 기준 아님 — 만료시각) }
upgrades: Record<string, number>; // 도구 업그레이드 단계

// ItemDef 확장
potion?: { category: 'gathering' | 'combat'; effect: BuffEffect; durationMs: number };
```

- 버프 만료는 `simulate`가 처리 (오프라인 중 만료 포함 — 만료 시각까지만 효과 적용해
  구간을 나눠 정산. 구현 단순화: 버프 경계로 elapsed를 분할해 simulate 내부 2-pass)
- 슬롯 해금 총 레벨(30/90) 재검토 — 스킬 8종이면 총 레벨 인플레 발생.
  v4 마이그레이션에서 해금 기준 상향 조정 가능 (balance-report로 결정)

## 3. 작업 분해 (순서대로)

### 3-1. 낚시 + 요리 (음식 파이프라인)
- [ ] 낚시 액션 6종 (새우→청어→연어→…), 물고기 아이템
- [ ] 요리 액션: 물고기 → 음식 (`food.heal` 차등). **요리 실패 메커니즘은 넣지 않는다** (결정성·단순화)
- [ ] Phase 2 임시 음식(말린 고기)보다 상위 회복량으로 설계 — 요리할 이유
- [ ] 사냥 음식 슬롯과 연결 확인

### 3-2. 약초 + 연금술 (버프 파이프라인)
- [ ] 약초는 신규 채집 스킬 대신 **벌목/채광 부산물** (n% 확률 드랍 — 시드 난수 재사용)
- [ ] 연금술 액션: 약초+마정석 → 물약 6종 (채집 3종 + 전투 3종)
- [ ] 버프 시스템 코어 구현 (위 2번) + 버프 표시 UI (상단 바 아이콘 + 남은 시간)

### 3-3. 기존 스킬 티어 확장
- [ ] 벌목/채광/대장기술 액션을 스킬당 6~8개로 (레벨 1~99 구간 커버)
- [ ] 전투 몬스터도 상위 티어 2~3종 추가 (새 장비 티어와 페어링)

### 3-4. 상점
- [ ] `data/upgrades.ts` + `buyUpgrade` 커맨드 (골드 차감, 단계 증가)
- [ ] `cycleDurationMs`에 업그레이드 보정 적용
- [ ] 상점 패널 UI

### 3-5. 경제 검증
- [ ] balance-report에 "아이템 → 소비처" 매트릭스 출력 추가. 빈 칸 = 실패
- [ ] 시간당 골드 수입 vs 업그레이드 가격 곡선 점검

## 4. 테스트 계획
- 버프: 적용 중 사이클 단축 / 만료 후 원복 / **오프라인 중 만료 경계 정산** (핵심)
- 부산물 드랍: 시드 고정 시 결정성
- 업그레이드: 구매 → 속도 반영, 골드 부족 거부
- v3→v4 마이그레이션

## 5. 완료 기준 (DoD)
- [ ] 모든 아이템에 소비처가 있다 (매트릭스에 빈 칸 없음)
- [ ] 골드 사용처가 있고, 실제로 사고 싶다
- [ ] 슬롯 2~3개를 조합하는 의미 있는 선택이 생겼다 (낚시+요리 동시 가동 등)
- [ ] 신규 스킬 2종이 전투 성과에 체감되게 기여한다

## 6. 리스크 / 주의
- 콘텐츠 양산 단계라 지루할 수 있음 — 데이터 입력은 표(스프레드시트)에서 설계하고
  코드 생성/변환하는 흐름을 만들면 이후 Phase에서도 재사용 가능
- 버프의 오프라인 경계 정산이 이 Phase의 유일한 까다로운 코어 작업. 테스트 먼저 작성 권장

## 진행 기록

### 2026-06-12 — 코어 + UI 구현 완료 (3-1 ~ 3-5)
- 코어: 낚시·요리·연금술 3스킬(각 6/6/6액션), 약초 부산물(벌목·채광 12%, 액션별 누적
  사이클 순번 시드 — 전리품과 같은 원리), 물약 6종(채집 3 + 전투 3), 도구 업그레이드
  상점(도끼/곡괭이/낚싯대 5단계), 세이브 v4 마이그레이션
- 버프 오프라인 경계 정산: `simulate`가 버프 만료 시각으로 정산 구간을 분할(`settleSpan`)
  — 틱 누적과 일괄 정산의 결과 동일성을 테스트로 보장 (문서의 "2-pass" 대신 N-구간 분할로 일반화)
- 문서와 달라진 점:
  - **소비 작업의 재료 대기**: 재료가 떨어져도 부족한 재료를 전부 다른 활성 작업이
    공급 중이면(낚시→요리, 사냥→연금술) 중단하지 않고 대기한다 (`activelySupplied`).
    진행도는 재료가 없는 동안 쌓이지 않음 — 누적하면 재료 도착 순간 사이클이 일괄
    폭발해 틱 패턴 의존이 생긴다. `startAction`도 같은 판정으로 재료 0개 시작을 허용
    → "낚시+요리 동시 가동" DoD가 실제로 성립
  - 버프 만료 기준은 epoch 시각이라 **오프라인 중에도 시간이 흐른다** (게임시간 아님, 설계대로)
  - 기존 스킬 티어 확장: 벌목·채광 6액션(아다만타이트/오리할콘·마법나무/고대나무),
    몬스터 트롤(Lv38)·와이번(Lv48) — 와이번 비늘이 오리할콘 장비 재료
- 슬롯 해금 기준 40/100 → **45/110** (스킬 8종으로 시작 총 레벨 14→17 인플레 보정)
- balance-report에 아이템→소비처 매트릭스(빈 칸 검출)와 시간당 골드 수입 vs
  업그레이드 가격 곡선 추가. test-save에 `economy` 프리셋 추가
- 테스트 32 → 50개. 남은 것: DoD 수동 검증 (실플레이)
