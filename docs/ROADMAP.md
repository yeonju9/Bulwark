# 개발 로드맵

> 마지막 갱신: 2026-06-12
> 원칙: **각 Phase는 "플레이 가능한 상태"로 끝난다.** 다음 Phase는 이전 Phase를 직접
> 플레이해보고 배운 것을 반영해 시작한다. Phase 안에서 스코프를 늘리지 않는다 —
> 좋은 아이디어가 생기면 이 문서의 백로그에 적고 넘어간다.

## 전체 그림

```
Phase 0  스캐폴딩 + 슬롯 시스템          ✅ 완료
Phase 1  프로토타입 다듬기 (재미 검증)    ← 지금 여기
Phase 2  전투와 던전 (게임의 척추)
Phase 3  스킬 확장 (경제 순환 완성)
Phase 4  계정과 서버 저장 (서비스화 1단계)
Phase 5  폴리시 (상용 퀄리티 만들기)
Phase 6  소프트 런칭
Phase 7  라이브 운영 (시즌·거래소·월드보스)
```

각 Phase의 끝에는 **완료 기준(DoD)** 이 있다. 전부 체크되면 커밋·태그하고 다음으로 넘어간다.

Phase별 상세 구현 계획 (설계 결정 → 데이터 모델 → 작업 분해 → 테스트 → DoD):

## Phase 0 — 스캐폴딩 ✅ (2026-06-12 완료)

모노레포(core/web/server), 순수 시뮬 코어, 스킬 3종, 오프라인 정산,
localStorage 저장, 작업 슬롯 시스템(총 레벨 30/90 해금), 세이브 마이그레이션(v1→v2).

## Phase 1 — 프로토타입 다듬기 (재미 검증)

**목표**: "숫자가 올라가는 게 재미있는가"에 답한다. 신규 기능보다 손맛·피드백 위주.
**규모**: 작음 (며칠 단위)

### 기능

| 우선순위 | 기능 | 비고 |
|---|---|---|
| P0 | 레벨업/해금 토스트 알림 | 지금은 레벨업해도 아무 피드백이 없음. 재미의 핵심 |
| P0 | 밸런스 1차 패스 | 사이클 시간·XP·판매가 스프레드시트로 정리 후 조정. "다음 해금까지 N분" 곡선 설계 |
| P1 | 세이브 내보내기/가져오기 (JSON) | localStorage 유실 대비. 서버 저장 전까지의 안전망 |
| P1 | 진행 상황 초기화 버튼 (확인 2단계) | 테스트에도 필요 |
| P2 | 액션 카드에 "시간당 획득량" 표시 | 방치형 유저가 가장 원하는 정보 |
| P2 | 문서 타이틀에 진행 표시 (`[벌목 중] Idle RPG`) | 탭 전환 시 상태 인지 |

### 완료 기준 (DoD)
- [ ] 30분 연속 플레이에서 "다음 목표"가 끊기지 않는다 (해금 간격 체감 점검)
- [ ] 레벨업·슬롯 해금 시 시각적 피드백이 있다
- [ ] 세이브를 파일로 백업/복원할 수 있다

## Phase 2 — 전투와 던전 (게임의 척추)

**목표**: 채집→제작→**전투**→더 나은 채집의 순환 고리를 완성한다. RPG가 되는 단계.
**규모**: 큼 (수 주). 이 프로젝트에서 가장 중요한 Phase.

### 설계 (구현 전 결정할 것)
- 전투는 **스탯 기반 자동 전투** (실시간 조작 없음). 전투 스킬(공격/체력 등)도 다른 스킬처럼 XP로 성장
- 전투도 슬롯을 쓰는 하나의 "작업"인가? → 권장: 예. 기존 simulate 구조 재사용
- 사망 페널티 수위 (아이템 손실 없음 권장 — 방치형에서 하드 페널티는 이탈 요인)

### 기능

| 우선순위 | 기능 | 비고 |
|---|---|---|
| P0 | 캐릭터 스탯 (공격력/방어력/체력) | 장비+레벨에서 파생되는 순수 함수로 |
| P0 | 장비 슬롯 (무기/방어구) + 장착 UI | 제작한 검이 드디어 쓸모를 가짐 |
| P0 | 전투 시뮬레이션 (몬스터 사냥 = 액션) | core의 simulate에 전투 사이클 통합. 결정성 유지 — 의사난수(seed) 사용 |
| P0 | 몬스터 5~8종 + 전리품 테이블 | 전투로만 얻는 재료 (가죽, 마정석 등) |
| P1 | 던전 1개 (몬스터 연전 + 보스) | 클리어 보상: 다음 티어 제작 재료. 준비(음식/장비)가 전략이 되는 구조 |
| P1 | 음식 시스템 (전투 중 자동 회복) | Phase 3 요리 스킬의 수요처를 미리 만듦 |
| P2 | 도감 (처치한 몬스터/획득 아이템) | 수집 동기 |

### 기술 작업
- 전투 결과가 오프라인 정산에서도 동일해야 함 → seeded RNG (`lastTickAt` 기반 시드 등)
- 세이브 v3 마이그레이션 (장비/전투 스킬 추가)
- 전투 밸런스 헤드리스 시뮬레이터 (코어를 노드에서 돌려 승률표 뽑기 — 공유 코어 구조의 이점)

### 완료 기준 (DoD)
- [ ] 만든 장비를 끼고 던전 보스를 잡으면 다음 티어 재료가 나온다
- [ ] 준비 부족으로 전투에서 패배할 수 있다 (전략이 존재한다)
- [ ] 오프라인 중 전투 정산이 온라인과 같은 결과를 낸다 (테스트로 보장)

## Phase 3 — 스킬 확장 (경제 순환 완성)

**목표**: 스킬 3종 → 6종. 스킬끼리 서로의 산출물을 소비하는 경제 순환을 만든다.
**규모**: 중간. 시스템보다 콘텐츠 작업 위주.

### 기능

| 우선순위 | 기능 | 비고 |
|---|---|---|
| P0 | 낚시 (채집) | 요리 재료 공급 |
| P0 | 요리 (제작) | 물고기 → 음식 (전투 회복템). Phase 2의 음식 시스템과 연결 |
| P1 | 연금술 (제작) | 약초/마정석 → 물약 (버프: 채집 속도, 전투 스탯) |
| P1 | 각 스킬 액션 티어 확장 (스킬당 6~8개) | 레벨 1~99 구간을 채움 |
| P2 | 상점 (골드 사용처) | 골드가 쌓이기만 하는 문제 해결. 도구 업그레이드(채집 속도 +%) 판매 |

### 완료 기준 (DoD)
- [ ] 모든 스킬의 산출물에 소비처가 있다 (팔기만 하는 아이템 없음)
- [ ] 골드의 사용처가 있다
- [ ] 슬롯 2~3개를 조합하는 의미 있는 선택지가 생겼다 (예: 낚시+요리 동시 가동)

## Phase 4 — 계정과 서버 저장 (서비스화 1단계)

**목표**: "내 게임"에서 "서비스"로. 어느 기기에서나 이어서 플레이.
**규모**: 중간. 백엔드 본업 영역 — 빠르게 진행 가능할 것.

### 기능

| 우선순위 | 기능 | 비고 |
|---|---|---|
| P0 | 게스트 계정 (즉시 시작) → UUID 발급 | 가입 장벽 없이 시작, 나중에 계정 연동 |
| P0 | Google OAuth 연동 (게스트 → 정식 계정 승격) | |
| P0 | 서버 저장: PostgreSQL + 주기 동기화 | 기존 인메모리 SavesService 교체. 충돌 정책: 서버가 정산한 상태가 정본 |
| P0 | 서버 측 세이브 검증 강화 | simulate 재실행으로 "물리적으로 불가능한 진행" 거부 (이후 랭킹의 전제) |
| P1 | 다중 기기 동시 접속 처리 | 마지막 쓰기 승리 or 세션 잠금 — 단순하게 시작 |
| P2 | Rate limiting, 구조적 로깅 | 운영 준비 |

### 완료 기준 (DoD)
- [ ] PC에서 플레이 → 다른 브라우저에서 로그인 → 이어서 플레이
- [ ] 조작된 세이브(시간 조작, 아이템 주입)가 서버에서 거부된다
- [ ] 서버 재시작 후에도 데이터가 유지된다

## Phase 5 — 폴리시 (상용 퀄리티 만들기)

**목표**: "개인 프로젝트"와 "서비스되는 게임"의 차이를 메운다.
**규모**: 중간~큼. 지루하지만 퀄리티가 결정되는 단계.

### 기능

| 우선순위 | 기능 | 비고 |
|---|---|---|
| P0 | AI 생성 아이콘 세트로 교체 | 스타일 가이드 프롬프트 고정 → 일괄 생성 (Gemini/Claude 활용, 유료 에셋 없음). 데이터의 `icon` 필드를 이미지 경로로 전환 |
| P0 | 온보딩/튜토리얼 (첫 10분 가이드) | 신규 유저가 설명 없이 루프를 이해해야 함 |
| P0 | 사운드 (액션 완료, 레벨업, 전투) + 음소거 설정 | 무료 라이선스(CC0) 소스 활용 |
| P1 | 마이크로 애니메이션 (획득 숫자 팝업, 레벨업 연출) | |
| P1 | PWA (설치 가능, 오프라인 셸) | 모바일 RPG의 꿈과 연결되는 지점 |
| P1 | 반응형 모바일 레이아웃 | |
| P2 | 통계 화면 (누적 획득량, 플레이 시간) | |
| P2 | 다국어 준비 (ko 기본, en 추가) | 글로벌 방치형 커뮤니티가 큼 — 노출 기회 |

### 완료 기준 (DoD)
- [ ] 게임을 모르는 지인이 설명 없이 10분간 막히지 않고 플레이한다
- [ ] 모바일 브라우저에서 쾌적하게 플레이된다
- [ ] 이모지 플레이스홀더가 화면에 없다

## Phase 6 — 소프트 런칭

**목표**: 실제 유저의 피드백 루프를 연다.
**규모**: 작음 (기능보다 배포·운영 작업)

- 도메인 + 배포 파이프라인 (프론트: Cloudflare Pages, 서버: Fly.io/Railway급)
- 에러 트래킹(Sentry 무료 티어), 핵심 지표 (DAU, D1/D7 리텐션, 평균 세션)
- 공개 채널: itch.io 등록, 방치형 커뮤니티(r/incremental_games 등) 피드백 스레드
- 피드백 수집 창구 (인게임 링크 → Google Form이면 충분)

### 완료 기준 (DoD)
- [ ] 모르는 사람이 URL로 접속해 플레이하고 있다
- [ ] 에러와 리텐션을 대시보드로 볼 수 있다

## Phase 7 — 라이브 운영 (차별점 실행)

**목표**: 기획 단계에서 정한 차별점 — "방치형에 라이브 서비스를 1급 시민으로".
소프트 런칭 피드백에 따라 순서·내용이 바뀔 수 있음.

| 기능 | 비고 |
|---|---|
| 주간 월드 보스 | 서버 전체 누적 딜 + 기여도 보상. 크론 잡 하나로 도는 첫 라이브 콘텐츠 — 가장 먼저 |
| 랭킹 (스킬별/총 레벨) | Phase 4의 세이브 검증이 전제. Redis sorted set |
| 거래소 (비동기 아이템 매매) | 경제 설계 신중하게 — 인플레이션 관리, 수수료 싱크 |
| 시즌 리그 | 몇 달 단위 새 캐릭터 경쟁 + 시즌 보상 → 본 계정 영구 보너스 |

## 백로그 (지금 안 함)

Phase 진행 중 떠오른 아이디어는 여기에만 적고 현재 Phase를 끝낸다.

- 펫/동료 시스템
- 업적 시스템
- 프레스티지(환생) 메커니즘 — 시즌 리그와 역할이 겹치므로 보류
- 길드
- 수익화 (광고 제거, 치장) — 유저가 생긴 뒤에 고민

## 운영 원칙

1. **세이브 호환성은 깨지 않는다.** 스키마 변경은 반드시 버전 증가 + 마이그레이션.
2. **시뮬 코어의 결정성을 지킨다.** `Date.now()`/`Math.random()`을 코어에 넣지 않는다 (난수는 seed 주입).
3. **밸런스 수치는 데이터 파일에만.** 코드 로직과 분리 유지.
4. **Phase 중간에 스코프를 늘리지 않는다.** 백로그에 적는다.
